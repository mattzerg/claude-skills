# Q2.3 claude-mem worker hook — diagnosis

**Status:** Diagnosed; upstream issue, not a local OS rot.

## Symptom

PostToolUse:Write logged "claude-mem worker unreachable for 5 consecutive hooks" during Phase 1 P1.4 build (2026-05-31T03:27Z).

## Root cause

`claude-mem` plugin (`@thedotmack/claude-mem v13.4.0`) uses a Node-based long-running worker started via `bun-runner.js` → `worker-service.cjs`. On every PostToolUse it pipes the tool payload via stdin to the worker.

When stdin arrives empty (race condition, sub-shell handoff, or piped-stream issue) the runner emits **"[bun-runner] empty stdin payload received — issue #2188"** and exits with failure. After 5 consecutive failures, the plugin's failure counter trips the "worker unreachable" warning.

This is an UPSTREAM, KNOWN ISSUE — the plugin's own diagnostics name it (`issue #2188`).

## What this does NOT break

- Our Phase 1 / Phase 2 work is unaffected — the worker is a memory-augmentation layer separate from `~/.claude/projects/.../memory/MEMORY.md`. Writes still succeed.
- `learn-matt doctor` does not consult claude-mem.
- Decision-queue, miners, gates all run independently.

## Options (Matt to choose)

1. **Leave as-is** (current state). Periodic warnings in PostToolUse hook output; no functional impact. Recommended unless warnings become noisy.

2. **Disable the plugin's PostToolUse hook** by removing `claude-mem@thedotmack` from `~/.claude/settings.json:278` or marking it `false`. Stops all memory-augmentation features of the plugin (not just the failing hook). Aggressive.

3. **File upstream issue / patch the bun-runner.js** to retry empty-stdin or log gracefully. Best long-term fix but requires upstream coordination.
