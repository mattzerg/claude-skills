# Process Patterns

These patterns are derived from Matt's product-delivery, launch-planning, and scope-of-work materials.

## Pattern 1: Define the workstream, then the task list

The stronger delivery docs do not begin with random todos. They begin with:

- objective
- scoped workstream
- deliverables
- timing / phase
- dependencies

When streamlining a process, preserve that order.

## Pattern 2: Handoffs are usually the real bottleneck

Repeated drag tends to show up at:

- intake to execution
- execution to review
- review to publish / ship
- cross-functional state changes

Most process pain is not “too many steps.” It is ambiguous ownership at the boundary.

## Pattern 3: Distinguish recurring ops from one-off judgment

Good operating models separate:

- recurring, repeatable work
- exception handling
- strategy / judgment calls

Only the recurring layer should become rigid SOP or automation by default.

## Pattern 4: Use phase structure when the work is time-based

For launches, onboarding, implementation, and rollouts, time structure matters:

- pre-work
- execution window
- follow-up / stabilization

Do not flatten time-based workflows into a single checklist if sequence is load-bearing.

## Pattern 5: Every automation needs a trigger, owner, and fallback

Automation ideas are low quality unless they specify:

- trigger
- source of truth
- success state
- fallback path when the automation fails

## Pattern 6: Output should be operational

Default useful deliverables:

- current-state map
- future-state map
- owner table
- SOP
- automation brief
- KPI / review cadence
