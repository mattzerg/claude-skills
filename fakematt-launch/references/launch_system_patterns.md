# Launch System Patterns

These patterns are distilled from Matt's Google Drive product-delivery and marketing planning materials, including:

- `Accelerator Data Product and Content Brief`
- `Services Overview v3.1.0`
- `PMF Workbook`
- `Marketing Launch Planning Template`
- `Vinapa GTM Planning`
- `Vortexa Consulting Roadmap`

Use these as structural heuristics, not rigid templates.

## Pattern 1: Scope launches like deliverables

Matt's delivery docs repeatedly break work into:

- outcome
- scoped workstreams
- explicit deliverables
- rough timing or phase
- dependencies and open questions

For launch work, mirror that shape:

- launch outcome
- copy deliverables
- asset deliverables
- distribution workstreams
- proof/dependency gaps

## Pattern 2: Package copy, not just prose

The marketing planning files treat launch copy as a bundle:

- tagline
- short description
- long description
- USPs
- CTAs
- FAQ material
- channel variants

Default to building this bundle when the user says "help me launch this."

## Pattern 3: Pre-launch / day-of / post-launch

The launch sheets repeatedly segment work by timing, not just channel.

Always consider:

- what must exist before launch
- what needs active handling on launch day
- what follow-up matters after the spike

This is especially useful when the user has copy but no rollout sequence.

## Pattern 4: Missing assets are product risks

The copy-and-assets templates reveal that launches often fail because the message assumes assets that do not yet exist:

- GIF thumbnail
- screenshots
- one-pager
- logo variants
- social graphics
- support/contact details
- press-kit components

If the copy depends on one of these, flag it plainly as a launch blocker or gap.

## Pattern 5: Product delivery and marketing are one system

The briefs and roadmaps consistently connect:

- product readiness
- analytics/tracking readiness
- onboarding/support readiness
- CRM/comms readiness
- external narrative

When drafting a launch package, include operational gaps that would weaken the launch even if the prose is good.

## Pattern 6: Start with the wedge

The PMF and roadmap material repeatedly focus on:

- a specific user segment
- the narrowest compelling wedge
- the concrete habit or retention loop
- what is differentiated vs generic tooling

When the user's launch framing is broad, rewrite toward the narrow wedge first.

## Pattern 7: Keep the platform promise available

The vault also supports a broader Zerg story that should not get lost when the wedge is too narrow:

- solve hard, messy, cross-tool operational problems
- replace stacks of SaaS spend with a smaller custom system
- lower software cost materially when the current stack is bloated
- connect tools through shared data infrastructure instead of brittle glue
- implement AI in the business, not just add a chatbot to the surface

When the artifact is a solutions page, one-pager, services pitch, or platform narrative, include this broader layer explicitly.

## Pattern 8: Message ladder

For broad positioning, climb this ladder:

- immediate outcome
- differentiated mechanism
- economic consequence
- system consequence

Example shape:

- we solve the workflow that is currently broken
- we do it with one connected stack / shared runtime / robust data primitives
- that cuts software and integration cost
- it also makes AI implementation and cross-tool automation possible

Do not lead with all four at once in the first sentence, but do not collapse the message down to a single wedge if the sale is really strategic.
