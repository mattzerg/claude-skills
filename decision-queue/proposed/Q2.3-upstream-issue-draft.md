# Upstream issue draft — claude-mem worker empty-stdin race

**Status:** Drafted in Phase 4. File at `https://github.com/thedotmack/claude-mem/issues` when ready.

**Title:** `bun-runner: "empty stdin payload" race causes intermittent "worker unreachable" warnings`

---

## Body

### Summary

The `bun-runner.js` → `worker-service.cjs` chain occasionally receives an empty stdin payload, prints `[bun-runner] empty stdin payload received — issue #2188`, and exits non-zero. After 3–5 consecutive empty-stdin invocations the PostToolUse hook surfaces `claude-mem worker unreachable for N consecutive hooks`.

The warning is **cosmetic** — nothing else breaks — but it fires often enough to interfere with Claude Code session output.

### Repro

```bash
echo "" | node ~/.claude/plugins/cache/thedotmack/claude-mem/13.4.0/scripts/bun-runner.js \
  ~/.claude/plugins/cache/thedotmack/claude-mem/13.4.0/scripts/worker-service.cjs \
  hook claude-code observation
```

Output:

```
[bun-runner] empty stdin payload received — issue #2188
  script: /Users/.../scripts/worker-service.cjs
  payload byte length: 0
  payload type: null (no data event or stream error)
  platform: darwin
  shell: /bin/zsh
  stdin TTY: undefined
  timestamp: 2026-06-01T04:11:34.551Z
  CLAUDE_PLUGIN_ROOT: /Users/.../claude-mem/13.4.0
```

### Environment

- macOS Darwin 25.4.0
- Node.js (running via Claude Code's bundled runtime)
- Shell: `/bin/zsh`
- Claude Code CLI (latest as of 2026-06-01)
- `@thedotmack/claude-mem` v13.4.0 (also reproduces on v13.3.0)
- PostToolUse hook configured per the plugin defaults

### Observation pattern

The "consecutive failure count" oscillates 3 → 5 → 3 → 4 across sessions. This suggests intermittent reproduction — sometimes stdin arrives correctly, sometimes empty. Plausible causes:

- Race between hook subprocess spawn and Claude Code piping the tool payload into its stdin
- macOS-specific issue with how Claude Code's hook subprocess inherits its stdin (e.g., async file descriptor not yet flushed when child reads)

### What I'd find useful

Either:

1. A retry-empty-stdin mode in `bun-runner.js` (sleep 50ms + re-read), which would absorb the race silently
2. A `--quiet` flag that demotes the warning to a stderr trace without exit-non-zero, so PostToolUse counts don't trip
3. Documentation on how to disable the PostToolUse hook gracefully if a user finds the warnings noisy

Happy to test any patch on macOS 25.4 + Claude Code latest.

### What I tried locally

- Reinstalled v13.3.0 (same behavior)
- Confirmed `scripts/bun-runner.js` and `scripts/worker-service.cjs` paths resolve correctly via the hook command's `$_P` resolution chain
- Empty-stdin invocation reproduces the exact warning shape on demand
